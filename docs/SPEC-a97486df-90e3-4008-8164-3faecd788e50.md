# SPEC: Wire Dashboard Gate Pipeline/Agents Panel to Live Runtime State

**Story ID:** a97486df-90e3-4008-8164-3faecd788e50  
**Gate:** architect  
**Version:** 1.0  
**Status:** Ready for Implementation

## Executive Summary

Wire the Mission Control dashboard's Gate Pipeline and Agents panel to display real-time runtime state from the MC2 orchestration layer. Currently the pipeline section shows static mock data; this story makes it live.

## Background

### Current State
The dashboard (`/`) shows a "Gate Pipeline" card with static content:
```tsx
{['architect', 'ui-designer', 'implementer', 'reviewer-a', 'operator', 'reviewer-b'].map((gate, i) => (
  <Box key={gate}>
    <Box display="flex" justifyContent="space-between" alignItems="center">
      <Typography>{gate}</Typography>
      <Chip label="pipeline stage" ... />
    </Box>
  </Box>
))}
```

### Target State (OpenClaw Parity)
The OpenClaw monitor `/agents` section shows per-agent:
| Field | Description |
|-------|-------------|
| Agent | Agent name/role |
| Session | Active session ID |
| Last Event | Timestamp of last activity |
| Model | Model currently in use |
| In | Input tokens |
| Out | Output tokens |
| Cache | Cache tokens |
| Last Tool | Last tool called |
| Thinking | Last reasoning/thought |

For MC2, we need equivalent data per gate.

## Requirements

### 1. Active Story Per Gate
- Show which story is currently being worked on in each gate
- Display story title (truncated if needed)
- Show session ID for debugging
- Status indicator (active/idle/blocked)

### 2. Last Event, Model, Message Per Agent
- **Last Event:** Timestamp of last activity (from `RunSession.startedAt` or `StoryEvent`)
- **Model:** Model currently assigned/used (from `RunSession.model`)
- **Provider:** Provider for the model (from `RunSession.provider`)
- **Last Message:** Last agent message or summary (from OpenClaw session data via webhook)

### 3. Live Runtime Data (Not Static/Mock)
- Fetch from MC2 database (`RunSession`, `DispatchLock`, `Story`)
- Poll at interval (5-10 seconds) or use Server-Sent Events
- Show real status, not hardcoded "pipeline stage"

### 4. Parity Checklist
| OpenClaw Field | MC2 Equivalent | Implementation |
|----------------|-----------------|----------------|
| Agent | Gate name | Static mapping |
| Session | RunSession.id | From DB |
| Last Event | RunSession.startedAt | From DB |
| Model | RunSession.model | From DB |
| Provider | RunSession.provider | From DB |
| Last Message | Agent callback evidence | From callback |
| Status | RunSession.status + DispatchLock | From DB |

## Technical Design

### 1. New API Endpoint: `/api/v1/orchestration/runtime-state`

Returns live state for all gates:

```typescript
interface GateRuntimeState {
  gate: string;                    // 'architect' | 'implementer' | etc.
  status: 'idle' | 'active' | 'blocked';
  activeSession: {
    sessionId: string;
    storyId: string;
    storyTitle: string;
    startedAt: string;             // ISO timestamp
    model: string | null;
    provider: string | null;
    lastEventAt: string | null;
    lastMessage: string | null;
  } | null;
}

interface RuntimeStateResponse {
  gates: GateRuntimeState[];
  lastUpdated: string;
}
```

### 2. Dashboard Component Updates

**File:** `src/app/page.tsx`

- Replace static gate list with runtime data fetch
- Add polling/SSE for live updates
- Show active story per gate with link
- Display model/provider info

### 3. Data Sources

| Source | Endpoint/Query | Data |
|--------|---------------|------|
| Active sessions | `prisma.runSession.findMany({ where: { status: 'active' } })` | session, story, model, provider |
| Story metadata | `prisma.story.findUnique({ where: { id } })` | title |
| Dispatch locks | `prisma.dispatchLock.findMany()` | gate occupancy |
| Last events | `prisma.storyEvent.findMany({ orderBy: { createdAt: 'desc' } })` | timestamps |

### 4. Polling Strategy

- Client-side polling every 10 seconds (simple, sufficient for dashboard)
- Or: Server-Sent Events endpoint for push updates (future enhancement)
- Initial load: fetch runtime state on mount
